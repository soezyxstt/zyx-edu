# Bundle V1.1 — Programmatic Course Import

## Goal

```
bundle.json  ──►  importer.ts  ──►  db.insert()
```

Zero mapping. The JSON shape mirrors the database schema. The importer resolves
references, generates UUIDs, and fills defaults.

**V1.1 revisions are additive and backward-compatible with V1 bundles.**
Every V1 bundle is a valid V1.1 bundle.

## Revision 1 — Metadata Section

Add a top-level `metadata` block for version tracking, debugging, and audit.

### Schema

```typescript
interface BundleMetadata {
  /** Bundle schema version (this document = "1.1"). */
  schemaVersion: string;

  /** ISO-8601 timestamp of generation. */
  generatedAt?: string;

  /** Identifier of the tool or person that produced the bundle. */
  generatedBy?: string;

  /** Primary content language (ISO 639-1). */
  language?: string;

  /** Institutional course code, e.g. "MA1101". */
  courseCode?: string;

  /** Arbitrary key-value pairs for workspace-specific metadata. */
  [key: string]: unknown;
}
```

### Rationale

- `schemaVersion` lets the importer reject or adapt to future formats
- `generatedAt` / `generatedBy` aid debugging when bundles come from external workspaces
- `language` and `courseCode` are the most common fields needed by downstream tooling
- Open-ended `[key: string]` avoids pressure to add every possible field now

### Example

```json
{
  "metadata": {
    "schemaVersion": "1.1",
    "generatedAt": "2026-06-18T10:00:00Z",
    "generatedBy": "gemini-workspace/kalkulus-1",
    "language": "id",
    "courseCode": "MA1101"
  },
  "author": { ... },
  "course": { ... }
}
```

## Revision 2 — Website Materials Array

### Schema Constraint

The `websiteMaterials` table has:

- `chapterId` text FK — **no unique constraint**
- Comment on table: `(Single Material per Chapter, M:N links removed)`

At the **schema level** the database supports multiple materials per chapter.
At the **application level** the code assumes 1:1:

```typescript
// lib/material-storage.ts — destructures first row only
const [material] = await db.select().from(websiteMaterials)
  .where(eq(websiteMaterials.chapterId, chapterId));
```

### V1.1 Change

Replace the singular `websiteMaterial` field with an array `websiteMaterials[]`.

```diff
- websiteMaterial?: { ... }
+ /** Array allows future types: main, examples, formula-sheet, summary. */
+ websiteMaterials?: Array<{ ... }>;
```

### What the importer does

- Inserts every entry in the array (schema allows this)
- When `websiteMaterials` has **exactly one** entry: the existing app code continues
  to work (destructuring picks the only row)
- When there are **multiple entries**: new app code is needed to select the right
  material by `type` or `slug`

### Author guidance

For now, always supply exactly one entry per chapter (the main material).
Additional types (worked examples, formula sheets, summaries) are schema-ready
but require application-side support before they render in the UI.

```json
{
  "websiteMaterials": [
    {
      "title": "Limit Fungsi",
      "slug": "limit-fungsi",
      "canonicalMarkdown": "# Limit Fungsi\n\n..."
    }
  ]
}
```

## Revision 3 — Remove KO-ID Coupling From Markdown

### Problem

The current example markdown references database UUIDs directly:

```
:::concept {koId="..."}
```

At bundle-write time the author does not know the UUID. If the importer
generates different UUIDs on each run, the markdown produces dangling
references.

### V1.1 Solution: `ref` attribute with bundle-local identifiers

The bundle author writes human-friendly KO references. The importer rewrites
them to UUIDs during canonicalization.

#### How it works

1. Every KO in the bundle receives a **bundle-local identifier**:
   - Explicit `$id` if the author provides one
   - Auto-derived from `title` (slugified) otherwise

2. The markdown uses `ref="<identifier>"` instead of `koId="..."`
   ```markdown
   :::concept {ref="definisi-limit"}
   :::
   ```

3. The importer canonicalizes the markdown before storing:
   - Parses all block attributes looking for `ref="..."`
   - Resolves each identifier against the chapter's KO list
   - Replaces `ref="..."` with `koId="<generated-uuid>"`
   - Stores the rewritten markdown in `canonicalMarkdown`

#### Identifier uniqueness contract

| Scope | Guarantee |
|-------|-----------|
| Within a chapter | `$id` + slugified `title` must be unique |
| Across chapters | No constraint (different chapters can have identically named KOs) |
| Resolution scope | Resolved within the enclosing chapter only |

#### Author-facing format

```markdown
:::concept {ref="definisi-limit"}
Limit $f(x)$ saat $x \to c$ adalah $L$.
:::

:::formula {ref="teorema-limit-utama"}
$$ \lim_{x \to c} [f(x) + g(x)] = L + M $$
:::

:::example {ref="contoh-soal-1", difficulty="easy"}
**Problem**: Hitung $\lim_{x \to 2} x^2$.
**Solution**: Substitusi langsung: $2^2 = 4$.
:::
```

#### Why not other strategies

| Strategy | Rejected because |
|----------|------------------|
| `conceptName` in markdown | Not unique per chapter; changes break imports |
| `canonicalSlug` | Only identifies concepts, not individual KOs |
| No references (implicit ordering) | Brittle; reordering KOs breaks all materials |
| UUIDs in bundle | Bundle not human-writable; breaks git diff |
| `koId` preserved as-is | Dangling references on re-import |

## Revision 4 — Source Traceability

Add optional `source` metadata to any entity in the bundle.

### Rationale

When course content is generated externally (e.g. via AI workspaces, crawled
from PDFs, or converted from legacy formats), the content producer needs a
way to trace each piece back to its origin. This aids debugging, regeneration
decisions, and audit trails.

### V1.1 Change

Every entity type gains an optional `source` field:

```typescript
interface SourceInfo {
  /** Origin type: "gemini-workspace", "pdf-extract", "legacy-migration", "manual", etc. */
  type?: string;
  /** Original file name, URL, or workspace path. */
  file?: string;
  /** Row-level or page-level identifier within the source. */
  reference?: string;
  /** Free-form notes about how this content was produced. */
  notes?: string;
  /** SHA-256 of the source material before any transformation. */
  originalHash?: string;
}
```

### Where it attaches

```diff
course: {
+ source?: SourceInfo;
  chapters: Array<{
+   source?: SourceInfo;
    concepts: Array<{ ... }>;
    knowledgeObjects: Array<{
+     source?: SourceInfo;
      ...
    }>;
    websiteMaterials: Array<{
+     source?: SourceInfo;
      ...
    }>;
    flashcardSets: Array<{
      ...
      flashcards: Array<{
+       source?: SourceInfo;
        ...
      }>;
    }>;
  }>;
  knowledgeRelationships: Array<{ ... }>;
  assessmentSources: Array<{
+   source?: SourceInfo;
    assessmentObjects: Array<{
+     source?: SourceInfo;
      ...
    }>;
  }>;
}
```

### What the importer does

- When `source` is present: stores it in the entity's `metadata` JSON field
  under a reserved `_source` key (or in the entity's `description` for courses
  and chapters, which have no `metadata` column)
- When `source` is absent: no effect
- Never required — purely informational

### Example

```json
{
  "knowledgeObjects": [
    {
      "$id": "ko-def-limit",
      "title": "Definisi Limit",
      "conceptName": "Limit Fungsi",
      "content": "Limit $f(x)$ saat $x \\to c$ adalah $L$...",
      "type": "definition",
      "bloomLevel": "understand",
      "source": {
        "type": "gemini-workspace",
        "file": "chapter-01.md",
        "reference": "block-3",
        "originalHash": "a1b2c3d4e5..."
      }
    }
  ]
}
```

## Revision 5 — Assessment Authoring Experience

### Current V1 structure

```json
{
  "assessmentSources": [
    {
      "title": "UTS 2026",
      "category": "uts",
      "year": 2026,
      "semester": 1,
      "sourceMarkdown": "---\ncategory: uts\nyear: 2026\n...\n---\n## Soal 1\n...",
      "assessmentObjects": [
        {
          "questionType": "multiple_choice",
          "difficulty": 3,
          "applicationLevel": 2,
          "pattern": "direct_substitution",
          "reasoningType": "analytical",
          "estimatedSteps": 2,
          "options": ["9", "3", "6", "0"]
        }
      ]
    }
  ]
}
```

### Problems for human authors

1. **AI-inferred fields** (`pattern`, `reasoningType`, `estimatedSteps`, `applicationLevel`) are tedious to write by hand
2. **`sourceMarkdown`** duplicates what `assessmentObjects` already describes (two sources of truth)
3. **`sourceHash`** and **`canonicalQuestionHash`** are implementation details the author should not touch

### V1.1 Changes

#### 5a. Mark AI-inferred fields optional with defaults

```diff
assessmentObjects: Array<{
  questionType: string;
  difficulty: number;              // 1-10 scale
  options?: string[];              // required for multiple_choice
  questionMarkdown?: string;       // default ""
  answerMarkdown?: string;
- pattern: string;
- reasoningType: string;
- estimatedSteps: number;
- applicationLevel: number;
+ pattern?: string;                // default "general"
+ reasoningType?: string;          // default "analytical"
+ estimatedSteps?: number;         // default 1
+ applicationLevel?: number;       // default 1
}>;
```

The importer uses the provided defaults when these fields are absent.
Post-processing (the existing `assessment-extractor.ts` or a new pass) can
refine them later.

#### 5b. Keep `sourceMarkdown` as the canonical question source

The `sourceMarkdown` field is the **authoritative** representation of the
assessment content. The `assessmentObjects` array is a **structured index**
into that markdown.

For a human writing a bundle:

```json
{
  "assessmentSources": [
    {
      "title": "UTS 2026",
      "category": "uts",
      "year": 2026,
      "semester": 1,
      "sourceMarkdown": "---\ncategory: uts\nyear: 2026\nsemester: 1\n---\n\n## Soal 1\nTentukan limit $x \\to 3$ dari $x^2$.\nA. 9\nB. 3\nC. 6\nD. 0\n\n## Soal 2\nHitung turunan $f(x) = x^3$.\nA. $3x^2$\nB. $x^2$\nC. $3x$\nD. $x$",
      "assessmentObjects": [
        {
          "questionType": "multiple_choice",
          "difficulty": 3
        },
        {
          "questionType": "multiple_choice",
          "difficulty": 4
        }
      ]
    }
  ]
}
```

The author writes the questions once (in `sourceMarkdown`) and provides only
the minimum per-question metadata. The importer extracts `questionMarkdown`,
`options`, `answerMarkdown` from the source text if they are not provided
explicitly.

**Design principle**: when `sourceMarkdown` and `assessmentObjects` both
contain overlapping data, `assessmentObjects` wins (explicit > inferred).

#### 5c. Auto-compute hashes

| Field | Strategy |
|-------|----------|
| `sourceHash` | SHA-256 of `sourceMarkdown` |
| `canonicalQuestionHash` | SHA-256 of `questionMarkdown` + `answerMarkdown` + `options` |

Author never provides these.

## Revision 6 — Bundle Validation Rules

The importer MUST validate a bundle before inserting any rows. Validation
passes produce no side effects on the database.

### Rule set

| # | Rule | Severity | Check |
|---|------|----------|-------|
| V1 | `metadata.schemaVersion` is present | ERROR | Reject if absent |
| V2 | `metadata.schemaVersion` is ≤ max supported | ERROR | Reject if future version |
| V3 | `author.email` resolves to an existing user or can be created | ERROR | Fail if email invalid |
| V4 | `course.title` non-empty | ERROR | |
| V5 | `course.category` is a valid enum value | ERROR | Check against schema enum |
| V6 | At least one `chapter` in `course.chapters` | WARN | A course with no chapters is technically valid but probably a mistake |
| V7 | Every `chapter` has a non-empty `title` | ERROR | |
| V8 | Within a chapter, every `$id` is unique | ERROR | Duplicate `$id` causes undefined `$ref` resolution |
| V9 | Within a chapter, all `knowledgeObjects` have unique slugified titles | ERROR | Ensures `ref` resolution is unambiguous |
| V10 | Every `concept` in a chapter has a non-empty `canonicalSlug` | ERROR | |
| V11 | Every `concept` has at least one `localization` | WARN | A concept with no localizations is invisible to students |
| V12 | Every `knowledgeObject` references an existing `conceptName` within the same chapter | ERROR | `conceptName` must match a concept's `displayName` |
| V13 | Every `knowledgeObject` has a valid `type` and `bloomLevel` | ERROR | Check against allowed enum values |
| V14 | `websiteMaterials` entry `slug` is unique within the chapter | ERROR | Two materials with the same slug would conflict |
| V15 | Every flashcard `ko$ref` (if present) resolves to a KO `$id` in the same chapter | ERROR | |
| V16 | Every `knowledgeRelationship` has `sourceKo$ref` and `targetKo$ref` that resolve to KOs somewhere in the bundle | ERROR | Cross-chapter KO refs allowed |
| V17 | Every `assessmentSource` has a valid `category` | ERROR | Must be one of: tutorial, quiz, uts, uas, tryout |
| V18 | Every `assessmentSource` has `year` set | ERROR | |
| V19 | Every `assessmentSource` `sourceMarkdown` is non-empty | ERROR | |
| V20 | `assessmentObjects.questionType` is non-empty | ERROR | |
| V21 | `assessmentObjects.difficulty` is 1-10 | ERROR | |
| V22 | Cyclic `knowledgeRelationships` | WARN | Graph theory supports cycles; warn that student-facing graph may render oddly |
| V23 | Empty `chapters` array | WARN | |
| V24 | Chapter with no `knowledgeObjects` and no `websiteMaterials` | WARN | This chapter has no learning content |
| V25 | `assessmentObjects` count differs from question blocks in `sourceMarkdown` | WARN | Suggests mismatch between structured index and source text |

### Validation output

```
Bundle: kalkulus-1.json
Schema version: 1.1

ERRORS (3):
  [V12] Chapter "Limit Fungsi": KO "definisi-limit" references conceptName
        "Limit" but no concept with that displayName exists. Available:
        ["Limit Fungsi", "Teorema Limit"]
  [V16] knowledgeRelationship: sourceKo$ref "ko-nonexistent" does not
        match any KO $id in the bundle
  [V20] assessmentSource "UTS 2026", question 2: questionType is empty

WARNINGS (2):
  [V6] Course has no chapters
  [V23] Chapter "Turunan" has no knowledgeObjects and no websiteMaterials

Validation FAILED. No rows inserted.
```

### Validation pass/fail semantics

| Result | Importer action |
|--------|----------------|
| Zero errors | Proceed with insertion |
| One or more errors | Abort. Print summary. Exit code 1. |
| Warnings only | Proceed. Print warnings to stderr. Exit code 0. |

## Revision 7 — Importer Responsibilities

### Production-grade checklist

| Concern | V1 baseline | V1.1 addition |
|---------|-------------|---------------|
| UUID generation | `crypto.randomUUID()` for every PK | Same |
| FK resolution | Resolve `$ref` by `$id` or position | Same + markdown `ref` → `koId` rewrite |
| MTD creation | One per chapter (auto-generated title + content) | Same |
| Slug generation | `title.toLowerCase().replace(/\\s+/g, '-')` | Same |
| Hash generation | SHA-256 of content | Same + `sourceHash` from `sourceMarkdown`, `canonicalQuestionHash` from question content |
| Defaults | Omitted → schema default | Same |
| Timestamps | Omitted → `defaultNow` | Same |
| Insert order | 12-step sequence | Same |
| Transactions | Single `db.transaction()` | Same + batch inserts for large bundles (100+ KOs) to avoid SQLite parameter limits |
| Dry run | `--dry-run` flag | Same + validation-only mode (`--validate`) |
| Validation | Implicit | Explicit pass/fail with printed report (Revision 6) |
| Markdown canonicalization | None | Rewrite `ref="..."` → `koId="<uuid>"` |
| Source metadata storage | None | Store `source` block in entity `metadata._source` |
| Import idempotency | None | Optional `--upsert` mode: match by `source.originalHash` or `slug` to update existing rows instead of inserting duplicates |
| Error recovery | Rollback on any error | Same + print which entity caused the failure |
| Progress reporting | None | Print progress per chapter: `[3/10] Chapter "Turunan" — 5 KOs, 1 material, 12 flashcards` |

### Idempotent import (`--upsert`)

```bash
bunx tsx scripts/import-bundle.ts bundle.json        # fresh import
bunx tsx scripts/import-bundle.ts bundle.json --upsert  # update existing
```

When `--upsert` is set and a row with matching `slug` (or `source.originalHash`)
already exists in the database, the importer updates that row instead of
inserting a new one. This allows:

- Re-running the same bundle after fixing validation errors
- Re-importing a regenerated bundle (e.g. after content fixes)
- Seeding the same course structure into different environments

### Reference map construction

Before inserting, the importer builds a two-pass reference map:

```
Pass 1 — Collect:
  For every entity with $id:
    record { $id → (entity type, chapter index, entity index) }

Pass 2 — Resolve:
  For every $ref value found (in attributes, markdown, relationships):
    look up $id in the map
    replace with the generated UUID
```

This resolves references in any order, including forward references.

### MTD creation strategy

The importer creates one `masterTeachingDocument` per chapter (not one per
course). Rationale:

- Each MTD tracks its own version, status, and revision history
- A course-wide MTD would accumulate unrelated content changes
- The existing `websiteMaterials` FK to `mtdId` expects per-chapter MTDs

```typescript
const mtdId = randomUUID();
await db.insert(masterTeachingDocuments).values({
  id: mtdId,
  courseId,
  title: `MTD - ${chapter.title}`,
  markdownContent: chapter.source?.originalHash
    ? `# ${chapter.title}\n\n(Auto-generated from bundle ${chapter.source.file})`
    : `# ${chapter.title}\n\n(Auto-generated by Bundle Importer v1.1)`,
  version: 1,
  status: "active",
  createdById: resolvedAuthorId,
});
```

The markdown content is minimal — the actual teaching content lives in the
`websiteMaterials.canonicalMarkdown` field. The MTD exists to satisfy the FK
constraint and to provide a versioning anchor for future KO extraction.

## Full TypeScript Definition

```typescript
interface BundleV1_1 {
  metadata: {
    schemaVersion: string;             // "1.1"
    generatedAt?: string;
    generatedBy?: string;
    language?: string;
    courseCode?: string;
    [key: string]: unknown;
  };

  author?: {
    email: string;
    name?: string;
  };

  course: {
    title: string;
    category: string;
    description?: string;
    source?: SourceInfo;

    chapters: Array<{
      title: string;
      description?: string;
      source?: SourceInfo;

      concepts?: Array<{
        $id?: string;
        canonicalSlug: string;
        localizations: Array<{
          lang: "id" | "en";
          displayName: string;
          aliases?: string[];
        }>;
      }>;

      knowledgeObjects?: Array<{
        $id?: string;
        title: string;
        conceptName: string;
        content: string;
        type: KnowledgeObjectType;
        bloomLevel: BloomLevel;
        difficulty?: Difficulty;
        importance?: Importance;
        tags?: string[];
        source?: SourceInfo;
      }>;

      websiteMaterials?: Array<{
        title: string;
        slug?: string;
        canonicalMarkdown: string;
        structuredContent?: Record<string, unknown>;
        source?: SourceInfo;
      }>;

      flashcardSets?: Array<{
        title: string;
        source?: SourceInfo;
        flashcards: Array<{
          front: string;
          back: string;
          ko$ref?: string;
          explanation?: string;
          source?: SourceInfo;
        }>;
      }>;
    }>;

    knowledgeRelationships?: Array<{
      sourceKo$ref: string;
      targetKo$ref: string;
      type: RelationshipType;
    }>;

    assessmentSources?: Array<{
      title: string;
      category: AssessmentCategory;
      year: number;
      semester?: number;
      sourceMarkdown: string;
      source?: SourceInfo;
      assessmentObjects: Array<{
        questionType: string;
        difficulty: number;              // 1-10
        options?: string[];
        questionMarkdown?: string;
        answerMarkdown?: string;
        pattern?: string;                // default "general"
        reasoningType?: string;          // default "analytical"
        estimatedSteps?: number;         // default 1
        applicationLevel?: number;       // default 1
        source?: SourceInfo;
      }>;
    }>;
  };
}

// Supporting types

type KnowledgeObjectType =
  | "definition" | "formula" | "example" | "misconception"
  | "exercise" | "summary" | "objective" | "concept_overview";

type BloomLevel =
  | "remember" | "understand" | "apply" | "analyze"
  | "evaluate" | "create";

type Difficulty = "easy" | "medium" | "hard";
type Importance = "high" | "medium" | "low";

type RelationshipType =
  | "prerequisite" | "related" | "extends"
  | "example_of" | "misconception_of";

type AssessmentCategory =
  | "tutorial" | "quiz" | "uts" | "uas" | "tryout";

interface SourceInfo {
  type?: string;
  file?: string;
  reference?: string;
  notes?: string;
  originalHash?: string;
}
```

## Migration Notes V1 → V1.1

| Concern | V1 | V1.1 | Migration |
|---------|----|------|-----------|
| Schema version | Implicit | `metadata.schemaVersion: "1.1"` | Add top-level `metadata` block |
| Website materials | `websiteMaterial` (object) | `websiteMaterials` (array) | Wrap object in array `[{ ... }]` |
| Markdown KO references | `koId="..."` (UUID) | `ref="<bundle-id>"` (local identifier) | Replace UUIDs with `$id` values |
| Source traceability | Not supported | Optional `source` on any entity | Add where needed; none required |
| Assessment fields | All required | AI fields optional with defaults | Remove `pattern`, `reasoningType`, `estimatedSteps`, `applicationLevel` if not authored |
| Hash fields | `sourceHash`, `canonicalQuestionHash` could be provided | Always auto-computed | Remove from bundle; importer handles |
| Validation | Implicit | Explicit rule set | None (backward-compatible) |
| Import modes | `--dry-run` only | `--dry-run`, `--validate`, `--upsert` | None |
| Markdown canonicalization | None | `ref="..."` → `koId="<uuid>"` | None (importer handles) |

### Backward compatibility

Every V1 bundle is a valid V1.1 bundle. The importer:

1. Detects that `metadata.schemaVersion` is absent (or `"1.0"`)
2. Treats `websiteMaterial` (singular) as `websiteMaterials` with one entry
3. Treats bare `koId="..."` values as UUIDs (passes them through unchanged)
4. Uses V1 defaults for missing assessment fields

No V1 bundle needs modification to work with a V1.1 importer.
