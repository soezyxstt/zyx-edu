# Bundle V1.1.1 — Minimal Revision

This document describes only the changes from **V1.1 → V1.1.1**.

Everything else in the V1.1 proposal remains unchanged.

---

## Change 1: concept$ref on knowledgeObjects

### Motivation

V1.1 requires the author to write `conceptName` as a plain string. The
importer then matches it to a concept's `displayName` within the same chapter.
This coupling is fragile:

- Renaming a concept's `displayName` breaks all KO references silently
- Two concepts with the same `displayName` (in different localizations) create
  ambiguity
- The bundle author cannot distinguish between "reference this concept by ID"
  and "guess which concept I mean by name"

### V1.1.1 Change

Add `concept$ref` to `knowledgeObjects`. `conceptName` becomes optional and
acts as a backward-compatible fallback.

```diff
 knowledgeObjects?: Array<{
   $id?: string;
   title: string;
-  conceptName: string;
+  concept$ref?: string;        // explicit $id of a concept in the same chapter
+  conceptName?: string;        // fallback: matched by displayName (V1.0 compat)
   content: string;
   type: KnowledgeObjectType;
   bloomLevel: BloomLevel;
   difficulty?: Difficulty;
   importance?: Importance;
   tags?: string[];
   source?: SourceInfo;
 }>;
```

### Resolution order

```
1. If concept$ref is present:
   → Resolve to concept with matching $id in the same chapter
   → Derive conceptName from that concept's first localization displayName
   → Set conceptId to the resolved concept UUID

2. Else if conceptName is present:
   → Match conceptName against localizations.displayName within the chapter
   → Set conceptId to the matched concept UUID
   → conceptName stored as provided

3. Else:
   → Validation error: "knowledgeObject must specify concept$ref or conceptName"
```

### What gets stored in the database

Exactly the same as before:

| DB column | Source |
|---|---|
| `conceptId` | Resolved UUID (same resolution logic, different lookup key) |
| `conceptName` | Derived from `displayName` when `concept$ref` is used; author-supplied when `conceptName` fallback is used |

The database schema is **unchanged**.

### Example

```json
{
  "concepts": [
    {
      "$id": "c-limit",
      "canonicalSlug": "limit-fungsi",
      "localizations": [
        { "lang": "id", "displayName": "Limit Fungsi" }
      ]
    }
  ],
  "knowledgeObjects": [
    {
      "$id": "ko-def-limit",
      "title": "Definisi Limit",
      "concept$ref": "c-limit",
      "content": "Limit $f(x)$ saat $x \\to c$ adalah $L$...",
      "type": "definition",
      "bloomLevel": "understand"
    }
  ]
}
```

Importer resolves `concept$ref: "c-limit"` → finds concept `$id == "c-limit"`
→ reads first localization `displayName: "Limit Fungsi"` → stores
`conceptName = "Limit Fungsi"` and `conceptId = <generated-uuid>`.

### Why not deprecate conceptName entirely

V1.0 bundles use `conceptName`. Every V1.0 bundle is a valid V1.1 bundle, and
every V1.1 bundle is a valid V1.1.1 bundle. Removing `conceptName` would break
that chain.

## Change 2: Explicit chapter references on assessmentSources

### Motivation

V1.1 places `assessmentSources` at the course level with no way to link them
to chapters. The compatibility audit (v1.1, table `assessmentSourceChapters`)
identified this as a gap: the DB requires `assessmentSourceChapters` rows for
chapter-scoped queries, but the only way to populate them was by parsing YAML
frontmatter from `sourceMarkdown`.

### V1.1.1 Change

Two additions:

**2a. Add optional `$id` to chapters** (so they can be referenced).

```diff
 chapters: Array<{
+  $id?: string;
   title: string;
   description?: string;
   source?: SourceInfo;
   ...
 }>;
```

**2b. Add `chapters` field to `assessmentSources`**.

```diff
 assessmentSources?: Array<{
   title: string;
   category: AssessmentCategory;
   year: number;
   semester?: number;
+  chapters?: string[];            // chapter $id or chapter title
   sourceMarkdown: string;
   source?: SourceInfo;
   assessmentObjects: Array<{ ... }>;
 }>;
```

### Resolution order

Each entry in `chapters[]` is resolved independently:

```
For each entry in chapters[]:
  1. If the entry matches a chapter $id in the course → resolved
  2. Else if the entry matches a chapter title in the course → resolved
  3. Else → validation error

One row inserted into assessmentSourceChapters per resolved chapter.
```

### What gets stored in the database

| DB table | Source |
|---|---|
| `assessmentSourceChapters.assessmentSourceId` | UUID of the inserted `assessmentSource` |
| `assessmentSourceChapters.chapterId` | UUID of the resolved chapter |

The database schema is **unchanged**.

### YAML frontmatter note

The `sourceMarkdown` YAML frontmatter may still contain a `chapters:` list for
use by the `assessment-extractor.ts` pipeline. The bundle's `chapters[]` field
is for the **importer** — it populates `assessmentSourceChapters` without
parsing markdown. The two sources of truth are:

- `assessmentSources.chapters[]` → importer → `assessmentSourceChapters`
- `sourceMarkdown` YAML frontmatter → `assessment-extractor.ts` → concept/KO mappings

They can coexist. The importer does not touch the YAML frontmatter.

### Example

```json
{
  "chapters": [
    { "$id": "ch-limit", "title": "Limit Fungsi" },
    { "$id": "ch-turunan", "title": "Turunan" }
  ],
  "assessmentSources": [
    {
      "title": "UTS 2026",
      "category": "uts",
      "year": 2026,
      "semester": 1,
      "chapters": ["ch-limit", "ch-turunan"],
      "sourceMarkdown": "---\ncategory: uts\nyear: 2026\n---\n\n## Soal 1\n...",
      "assessmentObjects": [
        { "questionType": "multiple_choice", "difficulty": 3 }
      ]
    }
  ]
}
```

## Updated Validation Rules

### New rule V26 — concept$ref resolution

| # | Rule | Severity | Check |
|---|------|----------|-------|
| V26 | Every `knowledgeObject` must have either `concept$ref` or `conceptName` | ERROR | Both absent → reject |
| V27 | `concept$ref` must resolve to a concept `$id` within the same chapter | ERROR | Unresolved ref → reject |
| V28 | When `conceptName` is used as fallback, it must match at least one localization `displayName` in the chapter | ERROR | No match → reject |
| V29 | `conceptName` fallback must match exactly one localization (not ambiguous) | WARN | Multiple matches → warn, pick first |

### Modified rule V12

```diff
- V12: Every `knowledgeObject` references an existing `conceptName`
+ V12: (superseded by V26–V29)
```

### New rules V30–V31 — chapter references on assessmentSources

| # | Rule | Severity | Check |
|---|------|----------|-------|
| V30 | Every entry in `assessmentSources.chapters[]` must resolve to a chapter `$id` or chapter title in the course | ERROR | Unresolved → reject |
| V31 | `assessmentSources.chapters[]` should not contain duplicate entries | WARN | Duplicates produce redundant `assessmentSourceChapters` rows |

## Updated Importer Responsibilities

### New or modified responsibilities

| Responsibility | V1.1 | V1.1.1 |
|---|---|---|
| KO `conceptId` resolution | Match `conceptName` → `displayName` | `concept$ref` → `$id` first; fall back to `conceptName` → `displayName` |
| KO `conceptName` derivation | Always from author | Derived from localization `displayName` when `concept$ref` used |
| `assessmentSourceChapters` | Not populated | Insert one row per resolved chapter in `chapters[]` |
| Chapter `$id` scan | Not needed | Collect all chapter `$id` values during Pass 1 of reference map |

### Updated insertion order

```diff
  9. flashcard        (needs flashcardSet, optional KO)
 10. assessmentSource (needs course + user)
-11. assessmentObject (needs assessmentSource)
+11. assessmentSourceChapter  (needs assessmentSource + chapter)  ← ★ NEW
+12. assessmentObject (needs assessmentSource)
 13. knowledgeRelationship (needs source KO + target KO)
```

### Reference map expansion

Pass 1 of the reference map now collects chapter `$id` values in addition to
concept and KO `$id` values:

```
Pass 1 — Collect:
  For every entity with $id:
    record { $id → (entity type, chapter index, entity index) }

  ★ NEW: Also collect chapter $id values:
    "ch-limit" → { type: "chapter", index: 0 }
```

This enables `assessmentSources.chapters[]` refs to be resolved during Pass 2.

## Migration Notes V1.1 → V1.1.1

| Concern | V1.1 | V1.1.1 | Migration |
|---|---|---|---|
| Schema version | `"1.1"` | `"1.1.1"` | Update `metadata.schemaVersion` |
| KO concept reference | `conceptName: "Limit Fungsi"` | `concept$ref: "c-limit"` | Replace string with `$id` pointing to concept; importer still supports old format |
| Chapter identification | Implicit (by position) | Optional `$id` | Add `$id` to chapters that need referencing |
| Assessment-chapter links | YAML frontmatter only | Explicit `chapters[]` field | Add `chapters: ["ch-limit", ...]` to each source; YAML frontmatter keeps working for the extractor pipeline |

### Backward compatibility

Every V1.1 bundle is a valid V1.1.1 bundle:

- Missing `concept$ref` with present `conceptName` → uses V1.0/1.1 fallback
  resolution (match by `displayName`)
- Missing `chapters[]` on assessmentSources → `assessmentSourceChapters`
  remains empty (same behavior as V1.1)
- Missing `$id` on chapters → chapters cannot be referenced, but everything
  else works as before

No V1.1 bundle requires modification.
