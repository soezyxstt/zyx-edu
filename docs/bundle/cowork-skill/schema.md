# Zyx Course Bundle Schema Reference

This is the authoritative schema for authoring **one chapter** of a Zyx course outside the
website, using whatever source material you have (a PDF ebook chapter, a 60-slide deck, scanned
notes, past exam PDFs). The output is two JSON files that the Zyx importer can load directly:
a **Learning Bundle** and an **Assessment Bundle**.

Keep this file in your project folder alongside `material/`, `assessment/`, and `output/`. The
authoring skill reads this file as ground truth before writing anything.

---

## 0. Scope: one chapter, one course, per run

A bundle always covers **exactly one chapter** inside **one course**. Don't try to author a whole
course in one pass. If the source material covers more than one chapter's worth of content, split
it into multiple runs, one chapter at a time.

Two separate output files, because they have different update cycles and different failure modes
(a typo in one exam question shouldn't force re-touching the whole chapter's content):

- **Learning Bundle**: course/chapter shell, concepts, Knowledge Objects (KOs), the website
  reading material, flashcards, and within-chapter relationships. Reusable across re-imports
  (matched by stable IDs).
- **Assessment Bundle**: exam/quiz questions for chapters that already exist (i.e. a Learning
  Bundle for that chapter must be imported first). References chapters **by title string**, not
  by the Learning Bundle's internal `$id`.

---

## 1. Global rules (apply to every field, every file)

- **Brand name**: always `Zyx` (capital Z, lowercase yx). Never `ZYX`, never `zyx`.
- **No em dashes (—) or en dashes (–), anywhere.** Use a comma, semicolon, or hyphen (`-`) instead.
  This applies to titles, descriptions, KO content, questions, answers, everything.
- **Math**: LaTeX only, `$...$` for inline, `$$...$$` on its own line for block equations. Never
  `\( ... \)` or `\[ ... \]`.
- **No placeholders.** Never write "TBD", "Lorem Ipsum", "akan ditambahkan nanti", or similar. If
  the source material genuinely doesn't cover something, leave it out and say so in your final
  report rather than inventing filler.
- **Language**: standard academic Bahasa Indonesia (Ejaan Yang Disempurnakan) unless the course is
  explicitly English-medium. No exclamation marks in explanatory text.
- **Never fabricate content not present in the source.** Worked examples, numbers, and claims must
  trace back to the uploaded material or past exams. If you need an illustrative example the
  source doesn't provide, mark it clearly as supplementary in your report instead of presenting it
  as sourced.
- **Forbidden fields**: never generate autogenerated primary/foreign keys (UUIDs except
  bundle-local `$id`), timestamps (`createdAt`/`updatedAt`), hashes (`generationHash`,
  `sourceHash`, `canonicalQuestionHash`), vector IDs, or any student progress/mastery/streak data.
  These are computed by Zyx at runtime.

---

## 2. ID stability

Every `$id` must be deterministic, not arbitrary. Use this scheme so re-running the same chapter
later produces the *same* IDs for unchanged content (this is what lets the importer do incremental
updates instead of duplicating everything):

| Entity | Formula | Example |
|---|---|---|
| Concept | `con-[COURSE_CODE]-[CHAPTER_CODE]-[slug]` | `con-fi1101-ch01-perpindahan` |
| Knowledge Object | `ko-[COURSE_CODE]-[CHAPTER_CODE]-[slug]-[type-suffix]` | `ko-fi1101-ch01-perpindahan-def` |
| Flashcard Set | `fset-[COURSE_CODE]-[CHAPTER_CODE]` | `fset-fi1101-ch01` |
| Assessment Source | `asrc-[COURSE_CODE]-[category]-[year]-[semester]` | `asrc-fi1101-uts-2026-1` |

KO type suffixes: `concept_overview` -> `overview`, `definition` -> `def`, `formula` -> `form`
(or `form-01`, `form-02` if more than one), `example` -> `ex-01`, `ex-02`, ..., `misconception` ->
`mis`, `exercise` -> `prac-01`, ..., `summary` -> `sum`, `objective` -> `obj`.

`COURSE_CODE` and `CHAPTER_CODE` (e.g. `fi1101`, `ch01`) come from whoever is running the skill;
ask if not given.

## 3. Learning Bundle shape

```json
{
  "metadata": {
    "schemaVersion": "1.1.1",
    "generatedAt": "2026-06-18T10:00:00Z",
    "generatedBy": "cowork/fisika-1a",
    "language": "id",
    "courseCode": "FI1101"
  },
  "author": { "email": "admin@zyx.id", "name": "System Admin" },
  "course": {
    "title": "Fisika Dasar 1A",
    "category": "Sains",
    "description": "Mata kuliah fisika dasar semester 1.",
    "chapters": [
      {
        "$id": "ch-kinematics",
        "title": "Kinematika Satu Dimensi",
        "description": "Mempelajari gerak lurus dalam satu dimensi.",
        "concepts": [],
        "knowledgeObjects": [],
        "websiteMaterials": [],
        "flashcardSets": [],
        "knowledgeRelationships": []
      }
    ]
  }
}
```

`course.chapters[]` must contain exactly one entry. `knowledgeRelationships` lives **inside the
chapter object**, not at the course level (course-level cross-chapter relationships aren't
supported by the importer).

### Field reference

**Course**: `title` (required), `category` (required, e.g. `"Sains"`, `"Matematika"`,
`"Rekayasa"`), `description` (optional).

**Chapter**: `$id` (required, bundle-local), `title` (required, this exact string is what a later
Assessment Bundle will reference), `description` (optional).

**Concept** (`chapter.concepts[]`):
- `$id` (required, bundle-local, used by KOs)
- `canonicalSlug` (required): unique lowercase kebab-case, e.g. `perpindahan-dan-jarak`. Must be
  unique across the whole course, not just this chapter.
- `localizations` (required, array): at least one entry with `lang: "id"`, `displayName` (the
  human-readable name), `aliases` (array of synonyms/search terms, used for RAG matching).

A concept should be explainable in 1-2 paragraphs but substantial enough to stand alone. "First
Derivative Test" is a good concept. "Derivative of x^2" is too narrow (that's an `example` KO, not
a concept). Don't duplicate a concept that already exists elsewhere in the course; if unsure
whether one exists, name it consistently and let the importer's concept-resolution handle reuse.

**Knowledge Object** (`chapter.knowledgeObjects[]`):
- `$id` (required, bundle-local)
- `title` (required)
- `concept$ref` (required): `$id` of a concept declared in this same chapter
- `content` (required): self-contained markdown. See section 6 below, "Writing KOs Zyra can
  retrieve in isolation" — this directly affects tutoring quality.
- `type` (required): one of `concept_overview`, `definition`, `formula`, `example`,
  `misconception`, `exercise`, `summary`, `objective`
- `bloomLevel` (required): `remember`, `understand`, `apply`, `analyze`, `evaluate`, `create`
- `difficulty` (optional, default `medium`): `easy`, `medium`, `hard`
- `importance` (optional, default `medium`): `high`, `medium`, `low`
- `tags` (optional, array of strings)

Ordering: KOs are sequenced by their array position (1-based `learningOrder`), so order them the
way a student should encounter them.

**Recommended KO coverage per concept** (a concept with only one KO is underdeveloped):
- 1 `concept_overview` (required, plain-language entry point)
- 1 `definition` (required, formal statement)
- 0-3 `formula` (if the concept is mathematical/scientific)
- 1-2 `example` (highly recommended, worked applications)
- 0-2 `misconception` (common pitfalls, especially anything past exams reveal students get wrong)
- 1 `summary` (recommended)

**Knowledge Relationship** (`chapter.knowledgeRelationships[]`):
- `sourceKo$ref`, `targetKo$ref` (required, `$id`s within this chapter)
- `type` (required): `prerequisite`, `related`, `extends`, `example_of`, `misconception_of`

Rules:
- `prerequisite` is directional and must be acyclic (A requires B must never coexist with B
  requires A). Chain depth within one chapter should not exceed 2 levels (A -> B -> C, not deeper).
- `example_of`: only from an `example` KO to the `formula`/`definition` it demonstrates.
- `misconception_of`: only from a `misconception` KO to the `definition`/`concept_overview` it
  corrects.
- `extends`: from an advanced KO back to the intermediate/intro KO it builds on.

**Website Material** (`chapter.websiteMaterials[]`, normally exactly one entry):
- `title` (required)
- `slug` (optional, derived from title if omitted)
- `canonicalMarkdown` (required): the full chapter reading text, see section 5.

**Flashcard Set** (`chapter.flashcardSets[]`):
- `title` (required), `flashcards` (required array)

**Flashcard** (`flashcardSet.flashcards[]`):
- `ko$ref` (required): the KO this card tests
- `front` (required): tests exactly **one** atomic fact. Never double-barreled.
- `back` (required): the correct answer
- `explanation` (optional but strongly recommended): why the answer is correct, and what
  misconception a wrong answer would reveal

## 4. Volume targets (per chapter)

| Item | Target | Hard minimum (pilot-ready) |
|---|---|---|
| Concepts | 8-15 | 5+ |
| Knowledge Objects | 15-30 | 15+ |
| Flashcards | 20-40 | 20+ |
| Assessment questions | as many as the source actually contains | 10+ |
| Website materials | 1-2 | 1+ |

If the source material is unusually large (a 60-slide deck, a 40-page ebook chapter), the volume
targets above still apply per **concept**, not per source page count — don't pad with filler to
hit a number, and don't compress real content to fit a small number either. Let the source's
actual density determine the final count; report if you significantly exceed or fall short of
these ranges and why.

## 5. Website material: compiler blocks

`canonicalMarkdown` is not plain prose. It's prose **plus** custom triple-colon container blocks
that wrap and reference specific KOs. Every KO declared in section 3 must appear inside exactly
one matching block, referenced by `ref="<ko-$id>"`.

```markdown
:::concept {ref="ko-fi1101-ch01-perpindahan-def"}
Perpindahan adalah perubahan posisi suatu benda, dihitung dari titik awal ke titik akhir.
:::
```

### Block tags

| Tag | Required structure | Used for |
|---|---|---|
| `:::learning_objective {bloomLevel="..."}` | — | Stated objective, usually near the top |
| `:::concept {ref="..."}` | — | `concept_overview` KOs. Qualitative explanation only, **no primary equations** (put those in `:::formula`) |
| `:::definition {ref="..."}` | — | `definition` KOs |
| `:::formula {ref="..."}` | must contain a markdown variable table | `formula` KOs |
| `:::example {ref="..."}` | must contain `#### Problem` and `#### Solution` headings | `example` KOs |
| `:::misconception {ref="..."}` | must contain `#### Misconception` and `#### Correction` headings | `misconception` KOs |
| `:::exercise` | — | `exercise` KOs |
| `:::summary` | — | `summary` KOs |
| `:::objective` | — | `objective` KOs |
| `:::engineering_insight {discipline="..."}` | — | applied/discipline-specific context, optional |
| `:::glossary_term {term="..."}` | — | inline term definitions, optional |
| `:::note {collapsible="true"}` / `:::warning {collapsible="true"}` | — | asides, optional |

### Visual blocks (use these, don't just describe visuals in prose)

If the source material includes a plotted function, a process/flow, or a labeled diagram, **you
must represent it with one of these blocks**, not a prose-only description. This is a common
failure mode of prior authoring passes: defaulting to text when the source clearly contains a
figure.

**`:::graph`** (mathematical function plot, YAML body):
```yaml
functions: ["y = sin(x)"]
domain: { min: -10, max: 10 }
samples: 200
```
Must contain at least one plottable function.

**`:::chart`** (data visualization, YAML body):
```yaml
chartType: line   # line | bar | scatter | pie
xLabel: "Waktu (s)"
yLabel: "Posisi (m)"
data: [{ x: 0, y: 0 }, { x: 1, y: 5 }]
```
Must contain at least one data point.

**`:::diagram`** / **`:::flowchart`** (process flow, edge syntax):
```
Source --> Target
Source -- Label --> Target
Source --> |Label| Target
```
Node IDs are derived automatically from edge endpoint names. Prefix with `"Step N: "` or
`"Langkah N: "` to let the compiler extract step ordering. `diagram` needs at least 1 node + 1
edge; `flowchart` needs at least 2 nodes.

### Formatting rules

- Never leave a `:::` block unclosed; this breaks the entire document's parsing.
- Don't stack compiler blocks back to back with no prose between them. Write 2-3 paragraphs of
  connective narrative, motivation, or intuition between blocks so the chapter reads like a
  textbook, not a list of tagged fragments.
- A block's `ref` must point to a KO `$id` declared in this same chapter. Cross-chapter references
  are not supported.
- Don't put primary equations inside `:::concept`; use `:::formula`.

## 6. Writing KOs Zyra can retrieve in isolation

Knowledge Objects are embedded and retrieved individually by the AI tutor (Zyra). A KO that
depends on surrounding context for meaning will retrieve badly and confuse students. For every KO:

- **Self-contained**: don't assume the reader saw any other KO. Name the law/theorem/process
  explicitly inside the KO text itself.
- **Define every symbol locally**: if a KO contains a formula, restate what every variable in it
  means, inside that same KO ("di mana $F$ adalah gaya, $m$ adalah massa, dan $a$ adalah
  percepatan").
- **No relative references**: never write "seperti rumus di atas," "dari sini," or "lihat diagram
  di bawah." KOs are retrieved as flat, unordered blocks; there is no "above" or "below."
- **Include synonyms**: mention common alternative terms a student might actually type when
  asking the tutor a question.

## 7. Assessment Bundle shape

```json
{
  "metadata": {
    "schemaVersion": "1.0",
    "generatedAt": "2026-06-30T10:00:00Z",
    "generatedBy": "cowork/fisika-1a"
  },
  "author": { "email": "admin@zyx.id", "name": "System Admin" },
  "course": { "title": "Fisika Dasar 1A" },
  "assessmentSources": [
    {
      "title": "UTS Fisika Dasar 1A - 2026",
      "category": "uts",
      "year": 2026,
      "semester": 1,
      "chapters": ["Kinematika Satu Dimensi"],
      "sourceMarkdown": "Soal 1. ...",
      "assessmentObjects": [
        {
          "questionType": "multiple_choice",
          "difficulty": 5,
          "options": ["A. 10 m", "B. 20 m"],
          "questionMarkdown": "Sebuah benda bergerak ...",
          "answerMarkdown": "Langkah 1: ...",
          "pattern": "direct_computation",
          "reasoningType": "procedural",
          "estimatedSteps": 2,
          "applicationLevel": 1
        }
      ]
    }
  ]
}
```

**Critical**: `assessmentSources[].chapters[]` must be the chapter's exact **title string** as
written in the Learning Bundle's `chapter.title` field (e.g. `"Kinematika Satu Dimensi"`), not the
bundle-local `$id` (`"ch-kinematics"`). The Assessment Bundle is a separate file with no access to
the Learning Bundle's internal ID namespace. Get this string wrong and the import fails outright.

### Field reference

**Assessment Source**: `title` (required, stable match key for re-imports), `category` (required:
`"quiz"`, `"tutorial"`, `"uts"`, `"uas"`, `"tryout"`), `year` (required, integer), `semester`
(optional, integer), `chapters` (array of title strings), `sourceMarkdown` (required, raw exam
text), `assessmentObjects` (required array).

**Assessment Object**:
- `questionType` (required): `"multiple_choice"` or `"short_essay"`
- `difficulty` (required, integer 1-10)
- `options` (required for multiple choice; array of strings prefixed with letters, e.g.
  `["A. 10 m", "B. 20 m"]`)
- `questionMarkdown` (required)
- `answerMarkdown` (required): full step-by-step derivation to the final answer, not just the
  answer itself
- `pattern` (required): `"direct_computation"`, `"graph_interpretation"`, `"proof"`,
  `"parameter_analysis"`, `"modeling"`
- `reasoningType` (required): `"procedural"`, `"conceptual"`, `"analytical"`
- `estimatedSteps` (required, integer, typically 1-4)
- `applicationLevel` (required, integer, typically 1-2)

**Zero loss principle**: every question present in the source exam document must produce one
`assessmentObjects` entry. Before writing the JSON, enumerate every question number found in the
source (a literal numbered list) and use that list as the count you must match. This is checked
programmatically in the authoring pipeline, not just asserted.

**Distractor quality**: wrong options should map to a plausible misconception, not be random or
lazy ("semua jawaban di atas benar" / "tidak ada yang benar" are not acceptable distractors).

## 8. Validation checklist

Before a bundle is considered done:

- [ ] `metadata.schemaVersion` is `"1.1.1"` (learning) or `"1.0"` (assessment)
- [ ] Learning Bundle has exactly one entry in `course.chapters[]`
- [ ] Every `knowledgeObject.concept$ref` resolves to a concept declared in the same chapter
- [ ] Every `flashcard.ko$ref` resolves to a KO declared in the same chapter
- [ ] Every `knowledgeRelationship` source/target `$ref` resolves within the same chapter, no
      cycles, prerequisite chains no deeper than 2
- [ ] Every KO `$id` appears in exactly one `:::` block inside `canonicalMarkdown`, with the
      correct required sub-structure (variable table for `formula`, Problem/Solution for
      `example`, Misconception/Correction for `misconception`)
- [ ] No unclosed `:::` blocks
- [ ] Zero `—` or `–` characters anywhere in either bundle
- [ ] All math uses `$`/`$$`, never `\( \)` or `\[ \]`
- [ ] Assessment Bundle's `chapters[]` strings exactly match the Learning Bundle's chapter title
- [ ] Assessment object count matches the enumerated question count from the source document
- [ ] No placeholder text anywhere
