# Zyx Assessment Bundle Specification (v1.0)

Status: Canonical contract for the "Assessment Bundle" half of the split bundle pipeline.

---

# Purpose

An Assessment Bundle carries exam/quiz content (`assessmentSources[]`, each with nested
`assessmentObjects[]`) for chapters of a course that has **already been imported** via a
[Learning Bundle](./learning-bundle-spec.md). It does not carry any course, chapter, concept, or
Knowledge Object content; those are exclusively the Learning Bundle's responsibility.

This lets an admin re-run just the exam-ingestion half of a course (e.g. add this semester's UTS)
without re-touching learning content, and vice versa.

---

# Why chapters are referenced by title, not `$id`

A Learning Bundle's `chapter.$id` (e.g. `"ch-kinematics"`) is a bundle-local identifier: it only
has meaning inside that one JSON document, resolved in memory while that bundle is being
imported. An Assessment Bundle is a **separate** file uploaded in a separate request, with no
access to that mapping. So an Assessment Bundle's `assessmentSource.chapters[]` entries must be
the chapter's **`title`** string as it exists in the database (i.e. exactly what you put in the
Learning Bundle's `chapter.title` field), not its `$id`.

If the referenced chapter title doesn't exist yet for the named course, the import fails with an
error telling you to import that chapter's Learning Bundle first.

# Shape

```json
{
  "metadata": {
    "schemaVersion": "1.0",
    "generatedAt": "2026-06-30T10:00:00Z",
    "generatedBy": "gemini-workspace/fisika-1a"
  },
  "author": { "email": "admin@zyx.id", "name": "System Admin" },
  "course": {
    "title": "Fisika Dasar 1A"
  },
  "assessmentSources": [
    {
      "title": "UTS Fisika Dasar 1A - 2026",
      "category": "uts",
      "year": 2026,
      "semester": 1,
      "chapters": ["Kinematika Satu Dimensi"],
      "sourceMarkdown": "Soal 1. ...",
      "assessmentObjects": [
        {
          "questionType": "multiple_choice",
          "difficulty": 5,
          "options": ["A. 10 m", "B. 20 m"],
          "questionMarkdown": "Sebuah benda bergerak ...",
          "answerMarkdown": "Langkah 1: ...",
          "pattern": "direct_computation",
          "reasoningType": "procedural",
          "estimatedSteps": 2,
          "applicationLevel": 1
        }
      ]
    }
  ]
}
```

`course.title` must match an existing course (created by a prior Learning Bundle import) exactly.
`assessmentSources` is a flat top-level array (no chapter nesting); each source declares which
chapters it covers via the `chapters` title array.

# Entity reference

## Course (reference only)

* `title` (string, **Required**): Must match an existing course's title exactly.

## Assessment Source

* `title` (string, **Required**): Title of the exam paper. Used as the stable match key for
  re-imports (mode `upsert`/`append`).
* `category` (string, **Required**): One of `"quiz"`, `"tutorial"`, `"uts"`, `"uas"`, `"tryout"`.
* `year` (integer, **Required**).
* `semester` (integer, Optional).
* `chapters` (array of strings, Optional): Chapter **titles** (see above), not `$id`s.
* `sourceMarkdown` (string, **Required**): Raw text of the exam paper.
* `assessmentObjects` (array, **Required**): Individual questions, see below.

## Assessment Object

* `questionType` (string, **Required**): `"multiple_choice"` or `"short_essay"`.
* `difficulty` (integer 1-10, **Required**).
* `options` (array of strings, **Required** for multiple choice).
* `questionMarkdown` (string, **Required**).
* `answerMarkdown` (string, **Required**): Step-by-step explanation.
* `pattern` (string, **Required**): `"direct_computation"`, `"graph_interpretation"`, `"proof"`,
  `"parameter_analysis"`, or `"modeling"`.
* `reasoningType` (string, **Required**): `"procedural"`, `"conceptual"`, or `"analytical"`.
* `estimatedSteps` (integer, **Required**): Typically 1-4.
* `applicationLevel` (integer, **Required**): Typically 1 or 2.

Same forbidden-fields rule as the Learning Bundle applies: do not include autogenerated IDs,
timestamps, or hashes (`canonicalQuestionHash` is computed by the importer if omitted).

# Incremental re-import semantics

Implemented in [lib/assessment-bundle-importer.ts](../../lib/assessment-bundle-importer.ts)
(`importAssessmentBundle`), shared by both the CLI and the admin Server Action:

- **Assessment Source** matched by `(courseId, title)`. `create` fails if it already exists,
  `upsert` updates it in place, `append` skips it.
- **Assessment Object** matched by `canonicalQuestionHash` (stable across reordering within the
  source): unchanged objects are left untouched, changed ones update in place, new ones insert.
  Objects no longer present in a re-uploaded source are **left in the database, not deleted**
  (the `assessment_objects` table has no retire/soft-delete column, unlike Knowledge Objects).
  The import result reports these as `aoOrphaned` for manual cleanup if needed.
- When `postProcess` is true, only **new or changed** assessment objects are re-mapped to
  concepts/KOs (heuristic keyword match against concept localizations/aliases, falling back to
  the referenced chapter's active KOs) and re-enqueued to the vector sync queue; unchanged
  objects keep their existing mappings. The course's assessment profile is recalculated once at
  the end via `updateAssessmentProfile`.

# Importer

- Shared core: [lib/assessment-bundle-importer.ts](../../lib/assessment-bundle-importer.ts)
  (`importAssessmentBundle`, `resolveCourseAndChaptersByTitle`)
